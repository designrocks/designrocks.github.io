The Module pattern

This pattern is used to mimic classes in conventional software engineering and focuses on public and private access to methods & variables. The module pattern strives to improve the reduction of globally scoped variables, thus decreasing the chances of collision with other code throughout an application.

The Revealing Module pattern

This pattern is the same concept as the module pattern in that it focuses on public & private methods. The only difference is that the revealing module pattern was engineered as a way to ensure that all methods and variables are kept private until they are explicitly exposed; usually through an object literal returned by the closure from which it's defined. Personally, I like this approach for vanilla JavaScript as it puts a clear emphasis on both the intent of the developer and the module itself.

Code Reuse patterns

The default pattern

Create an object using the Parent() constructor and assign this object to the Child()'s prototype.

Rent a constructor pattern

It borrows the parent constructor, passing the child object to be bound to this and also forwarding any arguments.

Rent and Set Prototype pattern

Restricts object creation for a class to only one instance.

Share the Prototype pattern

Restricts object creation for a class to only one instance.

A Temporary Constructor pattern

First borrow the constructor and then also set the child's prototype to point to a new instance of the constructor.

Klass pattern

Generally a pattern that should be avoided unless one is more comfortable with class than prototype.

Prototypal Inheritance pattern

Objects inherit from other objects.

GoF design patterns

Builder pattern

Constructs complex objects by separating construction and representation.

Factory method pattern

Creates objects without specifying the exact class to create.

Singleton pattern

Restricts object creation for a class to only one instance.

Decorator pattern

Dynamically adds/overrides behaviour in an existing method of an object.

Facade pattern

Provides a simplified interface to a large body of code.

Proxy pattern

Provides a placeholder for another object to control access, reduce cost, and reduce complexity.

Chain of responsibility pattern

Delegates commands to a chain of processing objects.

Command pattern

Creates objects which encapsulate actions and parameters.

Iterator pattern

Iterator accesses the elements of an object sequentially without exposing its underlying representation.

Mediator pattern

Allows loose coupling between classes by being the only class that has detailed knowledge of their methods.

Observer pattern

Is a publish/subscribe pattern which allows a number of observer objects to see an event.

Strategy pattern

Allows one of a family of algorithms to be selected on-the-fly at runtime.
